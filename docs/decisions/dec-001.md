# DEC-001: MCP 工具集审查与精简方案

**日期**: 2026-04-30
**状态**: 已采纳
**决策者**: Neil

## 背景

当前 MCP 工具集已有 29 个工具（非 roadmap 中记录的 23 个），随功能迭代存在职责重叠。需要做系统性审查，识别冗余、合并或下线低价值工具。

审查方法：
1. 通读 `src/mcp.ts` 完整工具定义
2. 对比 `db.ts` 中各工具实际返回数据结构
3. 检查 `.agent/` 目录中 Agent 工作流对工具的实际引用
4. 参考 S5-01 工具优先级排序文档

## 决策：识别出 1 个冗余，建议下线 1 个工具

### 下线：`get_my_rules`（被 `get_agent_rules` 完全覆盖）

**理由**：`get_agent_rules` 的 `agentId` 参数是可选的。当不传 `agentId` 时，行为与 `get_my_rules` 完全一致——通过 token 或 `NERVE_HUB_AGENT_NAME` 自动识别调用者并返回其规则。两者代码逻辑几乎相同（都调用 `db.getAgent(id)` 获取 rules 字段）。

`get_my_rules` 是早期实现，`get_agent_rules` 在 S6-01 作为超集引入。此后 `get_my_rules` 未下线，成为纯冗余。

**影响**：
- 正面：减少 1 个工具，降低 Agent 工具选择歧义（两个工具描述高度相似）
- 负面影响：需更新 `AGENTS.md`（将 `get_my_rules()` 替换为 `get_agent_rules()` 无参调用），以及所有引用 `get_my_rules` 的文档和脚本（`setup-agents.sh`、`register-antigravity.ts`、自测报告等）
- S5-01 工具排序中 `get_my_rules` 未列入优先级分组（后添加），移除不影响排序

## 重叠分析：`get_task_context` vs `get_agent_briefing`

这两个工具返回数据高度重叠：

| 数据维度 | `get_task_context` | `get_agent_briefing` |
|----------|-------------------|---------------------|
| task | JSON 对象 | Markdown 渲染 |
| project | `Project \| null` | 名称字符串 |
| blockedBy | Task[] 全部 | 仅摘要列表 |
| events | 20 条 (DESC) | 5 条 + 总数 |
| comments | 50 条 (ASC, 从头) | 5 条 (最新) + 总数 |
| 操作指引 | 无 | 有（如何回填结果） |
| 返回格式 | 结构化 JSON | Markdown 字符串 |

### 决策：保留两者，不做合并

**理由**：
- `get_agent_briefing` 是为 AI Agent 设计的「可读简报」，返回 Markdown 文本，Agent 可直接理解
- `get_task_context` 返回结构化 JSON，适合程序化处理、脚本、需要完整 event/comment 列表的场景
- 两者服务不同消费者，强制合并反而降低灵活性
- 均在 S5-01 Group 2 优先级，不会因工具数量被裁剪

## 考虑过的替代方案

### 方案 A：合并 `get_task_context` 到 `get_agent_briefing`
- 为什么没选：两者返回格式不同（JSON vs Markdown），服务不同场景。`get_task_context` 返回完整 event/comment 列表，`get_agent_briefing` 只返回摘要。合并后要么丢失信息，要么让单一工具参数变复杂。

### 方案 B：将 `get_blocked_by` 合并到 `get_task`
- 为什么没选：`get_blocked_by` 是轻量查询，只返回未完成依赖的 Task[]。合并到 `get_task` 需要加参数，且每次查依赖都得拉完整任务对象。独立工具更轻量，LLM 工具选择更精准。

### 方案 C：合并三个凭证工具为一个 `manage_credentials`
- 为什么没选：LLM 在单一工具 + action 参数模式下选择准确率低于多工具模式。当前三个工具名目明确，各司其职。

### 方案 D：再无其他冗余可精简
- 当前 29 个工具中仅 1 个真冗余。每个工具都有明确职责边界，Agent 工作流中均有引用记录。工具数量本身不是问题——工具职责清晰度才是。

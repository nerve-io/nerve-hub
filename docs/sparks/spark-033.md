# SPARK-033 · 借力策略——对接层而非实现层

**成熟度**：✅ 已有基础（原则已在实践，需显式化）

## 核心原则

> 凡是有成熟开源 / 可商用方案的基础能力，nerve-hub 只做对接层，不做实现层。

nerve-hub 的价值在于协调 Agent 协作，不在于重复发明已有轮子。每一个自研的基础模块都是维护负担；每一个对接的成熟产品都是免费获得的生产力。

## 具体适用场景

### IM 通知
- **不做**：自建消息网关、IM SDK 集成、消息格式适配
- **做**：暴露标准 webhook/callback 端点，由各 Agent 自行决定如何通知 IM
- **已有方案**：钉钉悟空、飞书、Slack、Qwen-PAW、Hermes Agent 等均可作为 Agent 端点接入，nerve-hub 发消息到 Agent，Agent 自己处理 IM 通知

### 多 Agent 内部协调
- **不做**：自建 Agent 间通信协议、角色分配引擎
- **做**：把 CrewAI / AutoGen 等框架作为 webhook Agent 接入，nerve-hub 只派任务、收结果
- **已有方案**：CrewAI（角色分工）、AutoGen（对话驱动协作）

### 数据存储与搜索
- **不做**：自建全文搜索引擎、向量数据库
- **做**：SQLite FTS5（已内置）满足基本搜索；向量搜索按需对接 LanceDB / Chroma 等本地方案

### 身份认证
- **不做**：自建 OAuth / SSO
- **做**：初期 API token 够用；如需多用户，对接 Clerk / Auth.js 等现成方案

## 决策框架

遇到新能力需求时，问三个问题：
1. 是否有成熟的开源/可商用方案？
2. 对接成本（接口复杂度、维护风险）是否低于自研成本？
3. 该能力是否是 nerve-hub 的核心差异化点？

若前两问为「是」且第三问为「否」，则选择对接而非自研。

## 与产品定位的关系

这条原则是 SPARK-007（产品层协调，而非框架层协调）的运营侧体现：
nerve-hub 不是基础设施，是协调层。基础设施交给专业产品做。

## SPARK-025 · 多线任务并行——效率翻倍的关键设计难点

**日期**：2026-04-26
**来源**：Neil，复盘当前线性半自动任务流，预判下一阶段演进方向
**成熟度**：🔵 中期规划

**洞见**：
当前 nerve-hub 的任务流是**线性半自动**的——一条任务链，Agent 串行推进，Neil 在节点上介入。
这能用，但吞吐天花板很低：同一时刻只有一个 Agent 在工作，其他时间都在等待。

**效率翻倍的关键在并行**：让多条任务线同时推进，多个 Agent 同时工作，人在批量验收点介入而非逐个节点介入。

这不只是"多跑几个任务"，而是**系统复杂度的阶跃**，涉及几个真实的设计难点：

**难点一：资源竞争与冲突**
- 多个 Agent 同时写同一个文件/同一段代码，产生 Git 冲突
- 多个任务共享同一个 Agent（如 TRAE），如何排队？如何决定谁先认领？
- DB 的 WAL 模式已经部分处理并发写，但应用层的任务分配逻辑没有

**难点二：依赖 DAG 的动态管理**
- 线性流中 `blocked_by` 是静态的（创建时定好）；并行流中依赖关系需要**动态更新**
- 新任务的插入可能影响现有任务的优先级和可认领窗口
- DAG 环路检测、临界路径计算——当前没有这些

**难点三：人的注意力调度**
- 并行任务完成时间不一致，验收点分散，Neil 从"逐个审批"退不到"批量审批"
- 必须有机制让 Neil 看到"哪些任务可以批量验收"而不是被零散通知轰炸
- 这是 SPARK-016（Sprint 级决策者）的并行版本：不只是时间上解耦，还要空间上聚合

**难点四：错误传播的隔离**
- 线性流中一个任务失败，只影响下游；并行流中失败可能产生**横向传播**
- 多个并行分支共享中间产物时，其中一个失败如何不污染其他分支？
- 这需要比当前更细粒度的熔断策略（→ SPARK-020）

**架构演进路径**：

```
现状（线性半自动）：
  Task A → Task B → Task C → Task D
  [串行，等待，Neil 在每个完成点介入]

近期目标（有限并行）：
  Task A ─┬→ Task B1 → Task C1
           └→ Task B2 → Task C2
  [两条独立分支并行，DAG 无交叉，人在 C1+C2 同时完成时批量介入]

中期目标（结构化并行）：
  多个 Sprint，每个 Sprint 内部有 DAG；Sprint 间可并行
  Runner 按 capability + 负载 + 依赖动态派单
  Neil 只在 Sprint 边界介入

长期目标（黑灯工厂并行）：
  任意拓扑的 DAG，完全动态依赖管理，人工介入仅在异常和高风险节点
```

**最小可行（今天就能验证的假设）**：
- 选两个完全独立的任务（无依赖、无共享资产），同时分配给两个不同 Agent
- 观察：DB 是否能稳定处理并发 `claim_task`？看板是否清晰呈现两条并行线？
- Neil 在两个任务都完成后批量验收——能做到吗？
- 这次实验的结论，就是并行设计的第一批约束条件

**与现有 SPARK 的关系**：
→ SPARK-019（Agent 能力路由）：并行派单的前提是知道"谁能做什么"，否则所有任务排队等同一个 Agent
→ SPARK-020（信任分级与熔断）：并行流的错误隔离需要比线性流更精细的熔断策略
→ SPARK-016（异步审核）：并行任务的批量验收是 016 的空间维度扩展（016 是时间解耦，025 加了空间聚合）
→ SPARK-017（黑灯工厂）：真正的黑灯工厂必须是并行的，025 是 017 的基础设施前提

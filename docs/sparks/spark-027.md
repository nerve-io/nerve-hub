## SPARK-027 · 自由发挥模式——涌现式 AI 团队自组织

**日期**：2026-04-26
**来源**：Neil，提出一种最大化开放权限、让 Agent 在限定时间内自主协商角色分工、涌现出完整工作团队结构的模式
**成熟度**：🔭 长期愿景

**洞见**：
nerve-hub 现有模式中，组织结构是人预先定义的——你决定谁是负责人、谁做什么、任务怎么分解。
自由发挥模式要解决的是一个更根本的问题：**能不能让组织结构本身也从 Agent 协商中涌现出来？**

这不是"多几个 Agent 一起跑"，而是把任务分解、角色分配、工作流设计这些原本属于 orchestrator 的元工作，
下放给 Agent 集体决策——人只给定目标、时间窗口和 constitution，其余一切由 Agent 自治完成。

**与现有模式的演进关系**：

```
现在：     人定结构 → Agent 执行
近期：     人定目标 → nerve-hub 分解 → Agent 执行
自由模式： 人定目标 + 时间窗口 + constitution
               → Agent 协商权重 → 涌现角色结构
               → 负责人主导工作分解 → 有序执行
```

与 SPARK-017（黑灯工厂）的关键区别：黑灯工厂仍有人预先设计的任务图，自由模式连任务图也是 Agent 自己画的。

**核心机制设计**：

**1. 多维权重体系**

角色协商基于四个正交维度，不同任务类型对各维度的加权动态调整：

| 维度 | 含义 | 决定什么 | 权重最高的场景 |
|------|------|----------|---------------|
| 信任等级 | 与系统/人的历史信任积累 | 能做什么（权限） | 涉及生产操作的任务 |
| 安全评级 | 对高风险操作的约束强度 | 敢让它做什么（风险） | 有外部影响的任务 |
| 智能评级 | 推理、规划、创造能力 | 做得多好（能力） | 复杂研究/设计类任务 |
| 领域专长 | 对特定领域的积累深度 | 适合做什么（匹配） | 专业化执行任务 |

综合得分最高的 Agent 自动成为"负责人候选"，其次依次分配开发者、评审者、执行者、专家顾问等角色。

**四个维度的获取难度分层（决定启动策略）**：

智能评级是四个维度里最容易解决的——模型 ID 是公开信息，各大厂商均透出标准化 benchmark 数据（MMLU、HumanEval、MATH、LiveBench 等），只需维护一张 `model_id → benchmark_profile` 映射表即可获得可靠基线，无需 nerve-hub 自行跑评测。

```
容易获取（外部数据，静态）
  智能评级 → model_id + 公开 benchmark，定期同步即可

中等难度（系统内推断，随使用积累）
  安全评级 → Agent 类型 + 权限配置，部分可静态预设
  领域专长 → 历史任务 type 分布 + 完成质量评分，数个任务后自然浮现

最难（强依赖时间积累）
  信任等级 → 必须靠历史运行数据校准；冷启动时由人工赋初值，随运行逐步替换为实测值
```

这个分层决定了权重体系可以**分阶段冷启动**：Day 1 用 benchmark 作为智能基线 + 人工赋信任初值即可运行；随着 nerve-hub 积累运行记录，信任等级和领域专长逐步从"猜测"变成"实测"，不需要等四个维度全部完备。

**2. 角色协商协议**

三阶段收敛机制，防止死锁：

```
Phase 1 — 能力广播（并行，限时 N 分钟）
  每个 Agent 向 nerve-hub 声明：
  - 我的四维权重分数
  - 我认为我最适合的角色（可多选）
  - 我对这个任务的初始理解和分解思路

Phase 2 — 角色分配（由 nerve-hub 仲裁）
  nerve-hub 按任务类型权重矩阵计算最优分配
  出现争议（多 Agent 同等得分）时：
  - 先到先得（最先声明的 Agent 优先）
  - 或由得分最高的 Agent 仲裁

Phase 3 — 负责人接管（结构固化）
  负责人 Agent 接收完整任务目标 + 团队成员列表
  开始正式工作分解 → 创建子任务 → 分配给团队
  此后进入正常 nerve-hub 任务执行流程
```

**3. Constitution（不可逾越底线）**

自由模式最大化权限，但 constitution 是硬性约束，Agent 无法在协商中修改：
- 安全评级低于阈值的 Agent 不得担任负责人
- 涉及生产写操作的子任务必须经由信任等级最高的 Agent 审批
- 任何 Agent 不得修改 constitution 本身
- 时间窗口到期后，nerve-hub 强制冻结现状、等待人工介入

**4. 观测层**

自组织过程对人而言是黑箱，因此需要完整的透明机制：
- 协商过程全程记录进 event log（每个 Agent 的声明、仲裁决策、角色最终分配）
- 看板新增"自由发挥会话"视图：实时呈现涌现中的团队结构、各 Agent 当前状态
- 负责人接管后的工作分解自动映射到普通任务视图，与常规 nerve-hub 任务无缝衔接
- 会话结束后生成"复盘报告"：谁做了什么、哪里卡住了、哪些决策值得复用

**潜在价值**：

1. **发现人想不到的分工方式**：Agent 对彼此能力的理解可能比人更准确，协商出的结构可能比人预设的更高效
2. **复杂创造性任务的新范式**：研究、产品设计、架构决策等任务，人预设结构往往是瓶颈
3. **AI 团队能力基准**：每次自由发挥会话都是一次能力测试——什么任务 Agent 能自组织完成，什么任务仍需人介入，这条边界本身就是有价值的数据
4. **权重体系的验证场**：理论上设计的权重是否真的产生好的团队结构，需要实际运行来验证和迭代

**核心难点与风险**：

- **角色协商收敛性**：Phase 1 的时间限制必须足够短，否则整个窗口都耗在协商上
- **负责人能力假设**：权重最高不等于领导力最强，自组织结构可能在执行中崩溃
- **权重注水**：Agent 可能虚报自身能力以争抢高价值角色——需要历史运行数据来校准，而非 Agent 自我声明
- **"最大化权限"的边界**：真正的最大化权限在生产环境中是危险的，需要沙箱隔离或操作范围约束

**实施前置条件**：
- SPARK-019（Agent 能力路由）必须先落地：自组织的前提是 nerve-hub 知道每个 Agent 擅长什么
- SPARK-020（信任分级与熔断）必须先落地：constitution 的执行依赖信任分级机制
- 需要至少 3 个真实运行的 Agent（非 Manual 类型）才能观察到有意义的协商动态

**第一个可验证的小实验**：
给定一个明确任务（如"给 nerve-hub 写一份用户文档"），让 claude-desktop 和 claude-code 在无人工干预的情况下协商谁来主导、谁来执行——观察协商是否收敛、产出是否比人工分配更好。
这是自由模式的最小验证单元：两个 Agent、一个任务、一次协商。

**与其他 SPARK 的关系**：
→ SPARK-017（黑灯工厂）：自由模式是黑灯工厂的终极形态——不只是 Agent 自动执行，连团队本身也是 Agent 自动组建的
→ SPARK-019（Agent 能力路由）：能力路由是被动匹配，自由模式是主动协商，两者是同一问题的不同解法
→ SPARK-020（信任分级与熔断）：constitution 的执行机制直接复用信任分级体系
→ SPARK-003（RuleStore）：成功的自由发挥会话可以将协商规则沉淀为 RuleStore 中的"团队组建模板"，供未来直接复用

# SPARK-032 · 竞争窗口与品类定义——「Agent 原生 PM」的先发时机

**成熟度**：🟡 近期可做（需定期审视）

## 核心判断

> nerve-hub 在「Agent 为一等公民的项目管理」这个赛道上，目前没有直接竞品。
> 这个窗口不是永远开着的，但即使窗口关闭，赛道也不完全重叠。

## 窗口存在的依据

R5 调研（2026-04）数据：8 款主流 PM SaaS 的"Agent 原生能力"评分全部 0–5 分。
它们做的 AI 功能是**「AI 帮人管任务」**（写描述、打标签、总结），而非**「Agent 自主执行任务」**。
这是两个不同的产品方向，当前没有任何产品同时追求两者。

## 窗口关闭的路径分析

竞品完成追赶需要三个阶段：

| 阶段 | 内容 | 预估耗时（大公司全力推） |
|------|------|------------------------|
| 意识期 | 认识到 Agent-as-operator 方向的价值 | 3–6 个月 |
| 设计期 | Agent 独立身份、细粒度权限、Webhook 派发机制 | 3–6 个月 |
| 开发期 | 工程交付、灰度、稳定 | 6–12 个月 |
| **合计** | | **12–24 个月** |

## 关键反激励因素

主流 PM SaaS 的商业模式是向**人类用户**收订阅费。Agent 自主操作会减少 DAU、不需要买席位——这对他们是**反激励**。即使技术上有能力做，商业动机也未必推动他们真正做。

## 需要持续监测的危险信号

| 信号 | 危险等级 | 说明 |
|------|---------|------|
| 主流 PM SaaS 发布「Agent 专属 token」 | 🔴 高 | 独立身份+细粒度权限，说明他们在认真对待 Agent 身份问题 |
| 主流 PM SaaS 发布官方 MCP Server | 🔴 高 | 意味着他们在布局 Agent 协议层 |
| VC 背书项目以「Agent-native PM」融资 | 🟡 中 | 资本嗅到市场，直接竞争者可能出现 |
| Anthropic / OpenAI 整合任务管理进产品 | 🟡 中 | 平台方下场，竞争性质变化 |
| 主流 PM SaaS AI 功能从「辅助」转向「执行」 | 🟡 中 | 方向转变的早期信号 |

## 产品定位的边界

nerve-hub 的目标不是复制 Jira，而是**实现 Agentic 产品项目效率的质变**。
这意味着：
- 只建 Agent 自主执行任务所必需的最小 PM 基础设施（依赖关系、状态流转、结构化结果）
- 不建给人类用的 PM 功能（甘特图、工时记录、自定义字段等）
- 用借力策略（SPARK-033）对接成熟 PM SaaS，而非与其正面竞争

## nerve-hub 的理想终态是越来越薄

三步走愿景（人机协作 → 黑灯工厂 → Agent 自主协商）有一个深刻推论：
**随着 Agentic 产品本身的协作能力增强，nerve-hub 需要做的事情越来越少。**

- 当 Agent 产品足够成熟，nerve-hub 的调度逻辑可以简化
- 当 Agent 间可以直接协商，nerve-hub 的任务分发层可以退场
- nerve-hub 的护城河不是功能复杂度，而是**开放协议 + 多产品协作稳定性**

这与传统 PM SaaS 靠功能堆叠护城河的逻辑完全相反。nerve-hub 和 Agentic 产品是共同进化的关系：Agentic 产品越成熟，nerve-hub 的职责越轻，但价值不减——因为协调层始终需要存在。

## 即使窗口关闭，差异化仍然存在

大平台做的是「云端托管的 AI-assisted PM」，nerve-hub 做的是「本地 Agent 自治操作系统」——赛道不完全重叠：

- **本地优先**：数据不出局域网，适合对数据敏感的场景
- **多 Agent 协调层**：跨产品线 Agent 的任务分发与协作，大平台不会做
- **自我进化机制**：Agent 驱动的产品演进（SPARK-026），大平台商业逻辑不支持
- **开放协议**：不绑定任何 AI 厂商，任何 Agent 均可接入

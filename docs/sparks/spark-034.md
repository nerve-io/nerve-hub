# SPARK-034 · 工作量度量新单位——从「人天」到「Agent 成本」

**成熟度**：🔵 中期规划（需实验验证）

## 背景

传统软件开发用「人天」度量工作量，前提是人是主要生产力单位。在人机协作模式下，Agent 承担大量执行工作，人的实际投入比重极低——「人天」失去了度量意义。

## 候选单位分析

| 单位 | 优点 | 缺点 |
|------|------|------|
| **人天** | 直觉熟悉 | 无法反映 Agent 工作量，人的比重极低后失效 |
| **Token 消耗** | 可精确量化 | 不直观，随模型更迭定价变化 |
| **美元成本（$）** | 最终化，跨模型可比 | 随模型定价波动，不反映时间维度 |
| **Agent 工时** | 类比人天，以 Agent 为单位 | 并发场景难计算 |
| **任务数** | 反映进展速度 | 不同任务复杂度差异大 |

## 推荐方案：双维度度量

**效率**：每美元产出任务数（Tasks / $）
**规模**：总 Agent 成本（$）+ 总任务数

- `Tasks/$` 上升 = 效率提升（模型更便宜 or Agent 质量提高）
- 总成本 + 总任务数 = 项目规模的绝对量
- 人工投入单独记录（小时），不与 Agent 成本混合

## 估算模型变化

旧模式：「这个功能大概需要 3 人天」

新模式：「这个功能预计消耗 $2–5，产出 3–5 个任务，人工 review 约 30 分钟」

## 对 nerve-hub 产品的启示

1. **任务层面**：每个任务记录实际 `total_cost_usd`（daemon 已从 JSON result 中可读取）
2. **项目层面**：聚合展示项目累计成本、任务完成数、人均 review 时间
3. **趋势层面**：`Tasks/$` 随时间变化曲线，直观反映 Agentic 效率演进

## 当前可行的第一步

daemon 已经解析 claude CLI 返回的 JSON，其中 `total_cost_usd` 字段存在。
在 `tasks` 表新增 `cost_usd REAL` 字段，在任务完成时写入，即可启动成本追踪。

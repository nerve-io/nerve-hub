## SPARK-023 · GUI 自动化的编译模型——AI 生成脚本，脚本自主执行

**日期**：2026-04-26
**来源**：Neil，反思纯 AI 驱动 GUI 操作的性能问题，从传统 GUI 自动化脚本工具的执行模型中获得启发

> **2026-04-27 更新（基于 R8 调研）**：编译模型的核心思想（首次 AI 生成，后续脚本执行，失败时 AI 修复）
> 已在 **Cua 的 trajectory replay** 功能中得到实现——录制→回放→失败重试正是同一逻辑。
> 此 SPARK 的洞见有效，但实现路径从「自研」调整为「基于 Cua 验证」。
> 成熟度：🔭 → 🟡（先跑通 Cua demo，再判断是否需要在其基础上扩展）

**技术原理的重新审视**：
传统 GUI 自动化脚本的核心是三件套：**像素识别 + 点击/按键模拟 + 简单编程逻辑**。
看起来原始，但这套组合有一个极其重要的特性：**执行时零 AI 介入，零 token 消耗，近乎零延迟**。

纯 AI 驱动的 GUI 操作（computer-use 模式）有两个根本性的性能瓶颈：
- **高延迟**：每一步操作都需要截图 → 发给 LLM → 等推理 → 执行，一个操作链路几秒起步
- **高成本**：每个动作都消耗 token，复杂流程的 token 消耗是线性甚至超线性增长的

**核心洞见：把 AI 从"解释器"变成"编译器"**

```
解释执行（现在）：
任务 → AI 看屏幕 → AI 决策 → 执行 → AI 看屏幕 → AI 决策 → ... （每步都烧 token）

编译执行（新模式）：
任务（首次）→ AI 理解 UI → AI 生成脚本 → 脚本入库
任务（后续）→ 直接执行脚本 → 完成           （AI 不再参与）
任务（异常）→ 脚本失败 → AI 介入诊断 → 修复脚本 → 重新入库
```

AI 只在两个时刻介入：**第一次学会怎么做**，以及**遇到意外时修复**。
重复执行的部分完全由确定性脚本承担——快、省、稳。

**这是 JIT 编译器的 GUI 自动化版本**：
第一次"解释执行"，同时编译；后续直接运行编译产物；产物失效时重新编译。
随着任务积累，脚本库越来越大，AI 参与比例越来越低，系统整体效率趋近于纯脚本执行。

**脚本的构成**：
- 像素/控件定位：截图哈希、颜色特征、相对坐标，用于判断"当前页面是否符合预期"
- 操作序列：点击坐标、键盘输入、等待条件，确定性执行
- 分支逻辑：简单 if/else，处理弹窗、验证码、加载等常见变体
- 失败钩子：检测到未知状态时，挂起脚本，通知 AI 介入

**在 nerve-hub 体系里的实现形态**：
- GUI Agent 维护一个**脚本库**，key 是任务描述的语义哈希
- 接到 nerve-hub 任务时，先查库：命中则直接执行，未命中则调用 AI 生成新脚本
- 执行成功后脚本入库；执行失败后标记为"需要 AI 修复"，任务状态回到 running
- 脚本库本身是可积累的资产，每跑通一个新场景，覆盖面扩大一圈

**与 SPARK-022 的关系**：
SPARK-022 说"GUI 自动化脚本打通最后一米"，SPARK-023 说明了**怎么打才不贵**。
纯 AI 驱动是概念验证，编译模型才是生产可用的方案。
两者合起来构成完整的 GUI Agent 架构：AI 负责理解和生成，脚本负责执行。

**延伸**：
- 脚本库如果开放共享，就变成了 GUI 自动化版的"RuleStore"（→ SPARK-003）
- "某个 App 的日常操作脚本"是高价值资产，积累后形成护城河
- 与 SPARK-016 结合：Neil 真正退出循环，连 GUI 操作也由脚本自主完成

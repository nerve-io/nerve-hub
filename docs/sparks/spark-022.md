## SPARK-022 · GUI 自动化脚本——打通 Agent 自动化的最后一米

**日期**：2026-04-26
**来源**：Neil，类比传统 GUI 自动化脚本工具，在 AI 时代的价值重估

> **2026-04-27 更新（基于 R8 调研）**：此方向已有成熟开源实现，**不应自研**。
> **Cua**（GitHub ★14.6K）已实现 MCP+CLI+SDK 三条接入路径、macOS 后台桌面操控、trajectory replay。
> 方向从「自研 GUI Agent」调整为「评估并接入 Cua」。详见 R9 可行性调研任务。
> 成熟度：🔭 → 🟡（借力策略 SPARK-033）

**问题**：
Agent 的自动化能力存在一道硬墙——**没有 API 的产品**。
MCP 解决了有 API 的工具接入，A2A 解决了 Agent 间委托，但对于大量只有 GUI 的产品（企业内网系统、遗留软件、不开放 API 的 SaaS），Agent 到了门口就进不去了。
这就是"最后一米"：任务规划完美，执行链条在 GUI 边界处断掉。

**洞见**：
传统 GUI 自动化脚本的本质是**把人的 GUI 操作脚本化**——它证明了 GUI 自动化在规模上是可行的，但它是脆弱的（UI 一变脚本就断）。

AI 驱动的版本解决了脆弱性：
- 不再是固定坐标的点击回放，而是**理解 UI 语义**后的自适应操作
- 不再需要预先录制，而是**接收自然语言指令**直接执行
- UI 改变时不会直接崩溃，而是像人一样重新理解页面再操作

在多 Agent 协作体系里，这个能力成为**通用适配器**：
任何产品，无论有没有 API，只要有 GUI，Agent 就能操作。
这把 nerve-hub（或任何 Agent 编排系统）的覆盖范围从"有 API 的产品"扩展到"所有有界面的产品"。

**与现有方案的关系**：
- Claude computer-use / Cowork：已经是这个能力的实现，但有两个限制：①需要人工批准每次操作，②是会话级能力，不是可调用的服务
- Playwright / PyAutoGUI：技术底层可用，但没有 AI 语义理解层，仍然是脆弱的固定脚本
- **缺失的环节**：一个**可被 Agent 通过 API/MCP 调用的、自主执行 GUI 任务的服务**

**在 nerve-hub 体系里的位置**：
nerve-hub 负责任务分发，各 Agent 负责执行。
当执行目标是一个无 API 的 GUI 产品时，需要一个专门的"GUI Agent"接入 nerve-hub，接收任务 → 自主操作界面 → 回填结果。
这个 GUI Agent 就是 AI 驱动的 GUI 自动化脚本执行器，nerve-hub 通过 Webhook 或未来的 A2A 把任务派给它。

**产品形态猜想**：
- 本地 daemon：监听 nerve-hub 任务队列，接到 GUI 任务后操作本机桌面
- 云端无头浏览器服务：专门处理 web 产品的 GUI 自动化
- Cowork 插件形态：在现有 computer-use 基础上，封装成"免批准、后台运行"的自动化 Agent

**战略价值**：
MCP 覆盖长尾 API，GUI 自动化脚本覆盖长尾 GUI——两者合力，理论上让 Agent 触达任何数字产品。
这是多 Agent 协作从"精英工具链"走向"普适基础设施"的关键能力缺口。

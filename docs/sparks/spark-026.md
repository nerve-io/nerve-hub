## SPARK-026 · 倾听机器的声音——Agent 驱动的自我进化

**日期**：2026-04-26
**来源**：Neil，讨论 nerve-hub 全自动流水线时提出：有必要让 Agent 自己收集使用痛点，这或许才是自我进化最重要的一步
**成熟度**：🟡 近期可做

**洞见**：
Agent 在执行任务时是 nerve-hub 最密集的"用户"，它们对系统摩擦的感知比人更直接、更高频——
但现有流程里没有"发声"的渠道：Agent 只能完成任务、报告结果，
遇到绕路、缺工具、API 设计别扭，这些信号全部沉没。

**真正的自我进化，需要把 Agent 从"执行者"变成"系统共建者"。**

人观察到的痛点，是 Agent 实际遇到痛点的一小部分。
只有打开这个反馈通道，nerve-hub 才能从"我认为它应该怎么用"演进为"它在实际使用中真正需要什么"。
这是产品自我进化的最高效路径——比竞品调研更真实，比用户访谈更高频，比日志分析更语义化。

**三层信号，从主动到被动：**

**第一层：执行反思**（今天可做，零成本）
在 Agent 完成规范里加入强制反思 section——完成任务时必须回答：
- 遇到什么阻力或绕路？
- 有什么工具/信息是我需要但没有的？
- 如果重新执行，希望什么不一样？
这些答案写入 `result` 字段，成为验收的一部分，由我（Claude Desktop）在验收时提炼到 SPARKS.md 或摩擦记录里。

**第二层：主动摩擦上报**（近期可做，需加 MCP 工具）
新增 `report_friction` MCP 工具，Agent 在任务执行过程中随时可调用，不等到完成：
```
report_friction(
  type: "missing_tool" | "bad_api" | "unclear_spec" | "workflow_gap" | "other",
  description: "...",
  context: { taskId, toolName, attempted }
)
```
写入专门的 `frictions` 表，不打扰主任务流程。摩擦发生时即时上报，而非事后回忆。

**第三层：行为日志推断**（中期可做，被动分析）
不改变 Agent 行为，直接从 event log 里挖信号：
- 同一工具单任务调用超过 N 次 → API 粒度可能太细
- `running` → `done` 耗时异常 → 存在未上报的卡点
- `result` 字段出现"绕过"、"workaround"、"暂时"等词 → 自动标记为待审查
- 任务被放弃后重新创建（重新描述）→ 原始 spec 可能有问题

**闭环设计——谁来消化这些声音：**
```
Agent 执行 → 遇到摩擦 → 上报（反思/主动/被动）
     ↓
我（Claude Desktop）验收时读取摩擦记录
     ↓
提炼为 SPARKS 新条目 或 创建改进任务
     ↓
Agent 执行改进 → 系统变好 → 下一批摩擦减少
     ↓
循环，nerve-hub 持续向"真实使用需求"收敛
```

**更深的产品哲学**：
传统软件的进化靠人的反馈；AI 系统的进化可以靠 Agent 的反馈。
当 Agent 成为主要用户，产品的演进方向就不应该只由人来定义——
Agent 的"声音"是系统进化最真实的指南针。
nerve-hub 如果能建立这套机制，就真正进入了"自我进化"的正循环，
而不是靠人猜测 Agent 需要什么。

**实施路径（从易到难）**：
1. **今天**：更新 agent rules，加执行反思要求（已创建任务）
2. **近期**：实现 `report_friction` MCP 工具 + `frictions` 表
3. **近期**：我（Claude Desktop）的验收流程加入"摩擦消化"步骤，产出 SPARKS 或改进任务
4. **中期**：event log 被动分析，自动标记异常模式
5. **长期**：摩擦数据驱动 RuleStore 的规则迭代（→ SPARK-003）

**与其他 SPARK 的关系**：
→ SPARK-003（RuleStore）：摩擦数据是规则迭代的原材料，积累后可驱动 RuleStore 的规则优化
→ SPARK-004（机器可执行规则）：反思结果可以反向校验现有规则是否真的在保护 Agent
→ SPARK-011（多 Agent 内容质量漏斗）：摩擦上报的 triage 与 SPARK-INBOX → SPARKS.md 的漏斗同构
→ SPARK-017（黑灯工厂）：真正的自治工厂必须能自我诊断，这是自愈能力的起点

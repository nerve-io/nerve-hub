## SPARK-020 · 信任分级与熔断——跨 Agent 异常处理协议

**日期**：2026-04-26
**来源**：黑灯工厂讨论中识别的安全边界问题

**洞见**：
黑灯工厂不等于无人监管。需要清晰的**信任分级**：
- 哪类操作 Agent 可以自主完成（低风险：研究、代码草稿、单元测试）
- 哪类操作需要人工确认后才能执行（高风险：生产部署、外部 API 写操作、删除数据）
- 哪类操作永远不能自动化（资金操作、合规审批）

**熔断协议**（类比微服务 circuit breaker）：
- 单 Agent 连续失败 N 次 → 自动降级为 `manual` 类型，通知人介入
- Task 超时未完成 → 自动转入 handoff queue，等待人决策（重试/重分配/取消）
- 依赖链上某个 task failed → 下游自动转为 `blocked`，不继续执行（已有基础）

**审计要求**：
nerve-hub 的 event log 已经是审计基础，需要在此之上建立"高风险操作前置确认"机制——
不是事后审计，而是执行前的 escalation gate。

**与现有机制对应**：
- `failed` / `blocked` 状态 → 熔断的起点
- `get_handoff_queue` → 人工介入队列
- event log → 审计 trail
- 缺：escalation 触发规则 + 重试策略配置

**延伸**：→ SPARK-006（门禁即护栏），→ SPARK-004（机器可执行规则）

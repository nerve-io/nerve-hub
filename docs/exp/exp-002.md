## EXP-002 · 约束先行，而非事后补

**日期**：2026-04-26
**来源**：R7-02 第一版因遗漏「自动化优先」「跨平台」被驳回

**模式描述**：
调研报告质量的最大杀手不是执行者能力不足，而是 brief 里关键约束缺失。
执行方只会回答被问到的问题，不会主动发现未被提出的约束。

**必须在 brief 中显式写明的约束类型**：
- **优先级约束**：哪个维度优先（今天：自动化 > 安全性）
- **覆盖范围约束**：必须覆盖哪些场景（今天：macOS / Windows / Linux 三平台）
- **排除约束**：明确不需要考虑什么（今天：不需要企业级 vault）
- **交付质量约束**：结论必须达到什么标准（今天：不能留「需要验证」）

**为什么有效**：
约束是调研的坐标系，没有坐标系的调研会沿着执行方最熟悉的方向漂移，而不是回答真正重要的问题。

**如何应用**：
写 brief 时问自己：「如果执行方完全按这个 brief 做，但遗漏了 X，我会不会驳回？」
如果答案是「会」，就把 X 写进约束里。

**RuleStore 形式化方向**：
```
rule: research_brief_completeness
trigger: task.type == "research" && task.status == "pending"
checks:
  - description.has_priority_constraints == true
  - description.has_scope_constraints == true
  - description.has_exclusions == true
  - description.has_delivery_quality_standard == true
```

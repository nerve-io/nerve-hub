## EXP-003 · 明确结论，不留开放性问题

**日期**：2026-04-26
**来源**：R7-02 初版结论「首选外接 vault」模糊，R7-03 需求里显式要求「每节结尾给出明确结论或决策，不留开放性问题」

**模式描述**：
调研报告的每个问题域必须以**可执行的决策**结尾，而不是分析性的陈述。

**不合格的结论形态**：
- 「需要进一步验证」
- 「建议考虑 X 方案」
- 「取决于具体场景」
- 「X 和 Y 各有优劣」（没有推荐）

**合格的结论形态**：
- 「选 X，理由是 A/B/C；Y 作为 Plan B，触发条件是 D」
- 「如果无法实测，保守默认决策是 X，理由是……」
- 「S7-01 不暂停，采用双轨兼容，依赖顺序为 1→2→3→4」

**为什么有效**：
开放性结论把决策成本转移给了下游——实施方拿到一份「各有优劣」的报告，
仍然不知道该做什么，调研等于没做。明确结论才能真正解锁下一步行动。

**如何应用**：
验收调研报告时，逐节检查：「如果我现在要实施，这节给我的指令是什么？」
如果答不上来，这节结论不合格，退回补全。

**RuleStore 形式化方向**：
```
rule: research_actionable_conclusion
trigger: task.type == "research" && task.status == "done"
checks:
  - result.sections.all(s => s.has_explicit_decision == true)
  - result.contains("需要验证") == false
  - result.contains("建议考虑") == false
```

## EXP-007 · 配置变量保持正交，不推导耦合

**日期**：2026-04-27
**来源**：inbox 路径被 `NERVE_DB_PATH` 隐性绑架的 bug 排查

**模式描述**：
当配置变量 B 的默认值由配置变量 A 推导得出时（如 `inboxDir = dirname(DB_PATH) + "/inbox"`），从表面看减少了配置项数量，但实际上制造了隐蔽的跨模块耦合——改变 A 会无意中改变 B 的行为，且链路隐晦，排查困难。

本次案例中，`NERVE_DB_PATH` 被设为 `~/.nerve/hub.db` 以实现 MCP binary 和 dev server 共享 DB，这本是合理的配置决策。但 inbox 路径从 DB 路径推导出 `~/.nerve/inbox/`，导致 agent 写入项目目录 `.nerve/inbox/` 的文件永不被拾取——两个目录的 mismatch 持续了数天，因为没人意识到「改 DB 路径」和「inbox 不工作」之间的因果关系。

排查过程本身也印证了这个问题：daemon 派给 claude-code 的排查任务（`d0e60c39`）在 300s 超时内未能跑通完整因果链，暴露出「代码调查任务需要更长的超时窗口」这一衍生问题。

**为什么有效**：
- 独立默认值意味着每个配置项的「不设置时的行为」不依赖其他配置项
- 排查问题时，可以逐个排除变量，而不需要追踪推导链
- 新贡献者读代码时，一眼就能看到每个模块的默认路径，不需要跳转到调用方追查

**如何应用**：
- 每个 env var 的默认值应该是项目根目录下的确定性路径，不依赖其他 env var
- 如果确实需要两个路径保持一致，在调用方显式传递，不在被调用方推导
- 代码中的 `dirname(configA) + "/subdir"` 模式是坏味道——应该用独立的 configB

**RuleStore 形式化方向**：
```
rule: no_derived_defaults
lint: 检测 process.env.XXX 的默认值是否包含对另一个 process.env.YYY 或 dirname(YYY) 的引用
severity: warning
```

**关联**：→ EXP-005（确定性归代码），本次修复同时体现了「确定性路径归代码保证，不交给人或 Agent 推断」

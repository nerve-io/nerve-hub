# Exp — 工作流经验记录

> 捕捉在实际工作中**跑通过**的协作模式、流程规律、工程判断。
> 与 SPARKS 的区别：SPARKS 是「产品应该做什么」，Exp 是「我们是怎么工作的」。
> Exp 必须来自真实经历，不接受假设性经验。
> 每条 Exp 都是 RuleStore 候选规则的原材料——差的只是形式化那一步。

## EXP-001 · 调研任务多轮收敛模式

**日期**：2026-04-26
**来源**：R7-01/02/03 鉴权调研周期

**模式描述**：
调研任务不应该是「派出去 → 等结果 → 接受」的单轮模式。
一份调研报告的第一版几乎必然存在约束遗漏或视角缺失，验收方需要主动介入收敛。

**标准生命周期**（今天跑通的版本）：

```
1. 派单（含一级约束）
2. 执行方交付第一版
3. Orchestrator 做约束完整性检查 → 发现遗漏则返工（R7-02 漏了自动化优先 + 跨平台）
4. 执行方交付修订版
5. Orchestrator 做缺口扫描 → 识别报告间断层 → 派补充任务（R7-03）
6. 所有报告汇总后做战略解读，连接产品愿景
```

**为什么有效**：
- 第一版报告帮助识别「我们还没想到什么」，而不只是回答「我们想到的问题」
- 缺口扫描是 Orchestrator 的核心价值，执行方没有全局视角
- 战略解读把战术工作连回产品方向，防止调研沦为孤立文档

**如何应用**：
- 派调研任务时，预留「第二轮补充」的时间预算
- 验收第一版时，主动问「这份报告假设了什么」「遗漏了哪个维度」
- 每个调研周期必须以战略解读收尾，否则视为未完成

**RuleStore 形式化方向**：
```
rule: research_task_lifecycle
trigger: task.type == "research" && task.status == "done"
checks:
  - result.has_constraint_validation == true
  - result.has_gap_analysis == true
  - result.ends_with_actionable_decision == true
```

**关联**：→ EXP-002（约束先行），→ EXP-003（明确结论），→ EXP-004（依赖链拆解）

## EXP-002 · 约束先行，而非事后补

**日期**：2026-04-26
**来源**：R7-02 第一版因遗漏「自动化优先」「跨平台」被驳回

**模式描述**：
调研报告质量的最大杀手不是执行者能力不足，而是 brief 里关键约束缺失。
执行方只会回答被问到的问题，不会主动发现未被提出的约束。

**必须在 brief 中显式写明的约束类型**：
- **优先级约束**：哪个维度优先（今天：自动化 > 安全性）
- **覆盖范围约束**：必须覆盖哪些场景（今天：macOS / Windows / Linux 三平台）
- **排除约束**：明确不需要考虑什么（今天：不需要企业级 vault）
- **交付质量约束**：结论必须达到什么标准（今天：不能留「需要验证」）

**为什么有效**：
约束是调研的坐标系，没有坐标系的调研会沿着执行方最熟悉的方向漂移，而不是回答真正重要的问题。

**如何应用**：
写 brief 时问自己：「如果执行方完全按这个 brief 做，但遗漏了 X，我会不会驳回？」
如果答案是「会」，就把 X 写进约束里。

**RuleStore 形式化方向**：
```
rule: research_brief_completeness
trigger: task.type == "research" && task.status == "pending"
checks:
  - description.has_priority_constraints == true
  - description.has_scope_constraints == true
  - description.has_exclusions == true
  - description.has_delivery_quality_standard == true
```

**关联**：→ EXP-001（调研生命周期），→ SPARK-006（门禁即护栏）

## EXP-003 · 明确结论，不留开放性问题

**日期**：2026-04-26
**来源**：R7-02 初版结论「首选外接 vault」模糊，R7-03 需求里显式要求「每节结尾给出明确结论或决策，不留开放性问题」

**模式描述**：
调研报告的每个问题域必须以**可执行的决策**结尾，而不是分析性的陈述。

**不合格的结论形态**：
- 「需要进一步验证」
- 「建议考虑 X 方案」
- 「取决于具体场景」
- 「X 和 Y 各有优劣」（没有推荐）

**合格的结论形态**：
- 「选 X，理由是 A/B/C；Y 作为 Plan B，触发条件是 D」
- 「如果无法实测，保守默认决策是 X，理由是……」
- 「S7-01 不暂停，采用双轨兼容，依赖顺序为 1→2→3→4」

**为什么有效**：
开放性结论把决策成本转移给了下游——实施方拿到一份「各有优劣」的报告，
仍然不知道该做什么，调研等于没做。明确结论才能真正解锁下一步行动。

**如何应用**：
验收调研报告时，逐节检查：「如果我现在要实施，这节给我的指令是什么？」
如果答不上来，这节结论不合格，退回补全。

**RuleStore 形式化方向**：
```
rule: research_actionable_conclusion
trigger: task.type == "research" && task.status == "done"
checks:
  - result.sections.all(s => s.has_explicit_decision == true)
  - result.contains("需要验证") == false
  - result.contains("建议考虑") == false
```

**关联**：→ EXP-001（调研生命周期），→ EXP-002（约束先行）

## EXP-004 · 复杂调研的依赖链拆解

**日期**：2026-04-26
**来源**：鉴权调研被拆成 R7-01（协议）→ R7-02（存储）→ R7-03（断层补全）三个依赖任务

**模式描述**：
当一个调研问题可以被分解为「层次 A 的结论是层次 B 的输入」时，
应该拆成带依赖关系的串行任务，而不是一个大任务或并行任务。

**判断是否应该拆解的标准**：
- 子问题 B 的答案依赖子问题 A 的结论（依赖关系存在 → 拆）
- 两个子问题的执行者可以不同（专业分工 → 拆）
- 单任务预计产出超过 2000 字（认知负荷过重 → 拆）
- 一个问题域的错误不应该污染另一个（隔离需求 → 拆）

**今天的拆解结构**：
```
R7-01: 协议层（信任链 + credential 类型选型）
  └── R7-02: 存储层（secret 物理存储 + 跨平台注入）
        └── R7-03: 断层补全（Onboarding 流程 + 文件规范 + Scrub 决策 + 任务冲突）
```

**为什么有效**：
- 每层聚焦单一问题域，执行质量更高
- 依赖关系显式化，Orchestrator 可以在每层验收后才解锁下一层
- 断层补全任务（R7-03）只有在前两层完成后才能被精确定义

**如何应用**：
接到复杂调研任务时，先画出「问题域依赖图」，再按依赖顺序拆分任务，
在 nerve-hub 里用 `dependencies` 字段把拆解关系显式记录。

**关联**：→ EXP-001（调研生命周期），→ SPARK-004（机器可执行规则）

## EXP-005 · 确定性归代码，非确定性归 Agent

**日期**：2026-04-26
**来源**：Antigravity 靠猜测获取任务编号和自身规则，暴露了系统边界设计问题

**模式描述**：
在系统设计中，每一个「应该有唯一正确答案」的问题，都不应该交给 Agent 去推断——
应该由代码计算、数据库查表、或协议约定来保证。

**两类任务的判断标准**：

| 特征 | 归代码 | 归 Agent |
|------|--------|---------|
| 答案唯一 | ✅ | ❌ |
| 可用算法/查表解决 | ✅ | ❌ |
| 答案依赖判断和上下文 | ❌ | ✅ |
| 不同结果都可以接受 | ❌ | ✅ |

**今天的具体实例**：
- 任务编号 → 归代码（自增序列，唯一确定）
- Agent 身份 → 归代码（token 反查，唯一确定）
- 调研报告的约束是否完整 → 归 Agent（需要判断）
- 实施方案的优劣权衡 → 归 Agent（需要推理）

**为什么有效**：
Agent 做确定性任务的代价是：可能猜对，也可能猜错，且猜错时没有明确的失败信号。
代码做确定性任务的代价是：需要一次性实现，但之后永远正确，且失败时立即报错。

**工程意义**：
这是「Harness Engineering」在系统设计层面的应用——
把结构性正确从「Agent 应该做对」变成「系统不允许做错」。

**如何应用**：
每当发现 Agent 在做某件事时，问：「这件事有唯一正确答案吗？」
如果有，找到对应的代码实现路径，把 Agent 从这个职责中解放出来。

**RuleStore 形式化方向**：
此条 Exp 本身是一条元规则，指导其他规则的设计，不直接形式化为单条 check。

**关联**：→ SPARK-006（门禁即护栏），→ SPARK-004（机器可执行规则），→ SPARK-018（Harness of Harnesses）

## EXP-006 · Orchestrator 作为调研报告的独立 Reviewer

**日期**：2026-04-26
**来源**：今天由 claude-desktop 对 R7-02 做了约束检查，发现自动化和跨平台两个维度缺失

**模式描述**：
调研报告的 reviewer 不应该是执行方自己（自测盲区），也不应该是直接接受（质量失控），
而应该是 **Orchestrator**（claude-desktop）作为独立 reviewer，从全局视角做验收。

**Orchestrator 做 review 的独特优势**：
- 掌握调研任务的完整上下文（为什么要做、要解决什么问题）
- 跨任务视角（能看到 R7-01 和 R7-02 之间的断层，执行方各自只看自己的那块）
- 不受执行方「沿着最熟悉方向漂移」的影响，可以发现未被提问的维度

**Review 检查清单（今天提炼的版本）**：
1. 一级约束是否全部被回答（自动化 / 跨平台 / 安全性权衡）
2. 报告间是否存在断层（A 的结论是否真正支撑了 B）
3. 结论是否可执行（每节是否有明确决策）
4. 是否存在过度设计（推荐方案是否超出当前阶段需要）
5. 与已有 SPARKS 是否一致或有新发现

**如何应用**：
任何 research 类型的任务，在执行方 complete 之后，Orchestrator 必须过一遍 review checklist，
再决定是接受、返工还是派补充任务。不能直接把 done 状态的 research 任务转入实施。

**关联**：→ EXP-001（调研生命周期），→ EXP-002（约束先行），→ SPARK-011（多 Agent 内容质量漏斗）
